# GitBiker 專案規範 (Gemini CLI)

本文件定義了 Gemini CLI 在此專案中的核心行為準則，其效力優於預設指令。

## 核心行為與語言規範

- **溝通語言**：一律使用**繁體中文**與使用者溝通。
- **規則映射**：嚴格遵循 `CLAUDE.md` 中定義的開發規範與準則。
- **Git 提交規範**：
  - 標題前綴使用英文（例如：`feat:`, `fix:`, `refactor:`, `docs:`）。
  - 提交訊息主體與詳細說明請使用**中文**撰寫。
  - 在執行 `git commit` 前，務必先顯示擬定的提交訊息並徵詢使用者確認。

## 開啟規劃模式 (Plan Mode)

- 當任務涉及系統架構調整、跨模組變更或使用者要求「規劃下一輪」時，必須先調用 `enter_plan_mode` 進行設計。
- 在規劃階段，若遇到邏輯分歧或需要決策的點，應主動暫停並跳出 prompt 與使用者溝通。

## 工程標準與代碼風格

- **Clean Code & Architecture**：遵循乾淨代碼與簡潔架構原則，嚴禁重複代碼 (DRY)。
- **一致性**：保持與既有代碼相同的設計風格、命名慣例與排版格式。
- **驗證**：所有變更必須包含對應的測試案例，並在完成後執行專案的 lint 與 type-check。

## 快捷關鍵字映射 (自動派發建議)

當使用者輸入以下關鍵字時，請視為特定的指令集，並引導至對應的分析流程：

- **站立會議 / 規劃下一輪 / 分析阻塞 / 跑回顧**：使用 `codebase_investigator` 進行專案狀態分析，模擬 Scrum Master 角色。
- **review**：針對最近的變更進行深入的代碼審查分析。
- **AI 評估 / 合規審查**：進行代碼質量掃描與架構合規性檢查。
